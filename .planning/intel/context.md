## TOPIC: Coding Standards and Conventions

source: CLAUDE.md

PHP Standards:
  - strict_types=1 in all PHP files
  - PSR-12 + Laravel conventions
  - Type hints on all parameters and return types
  - Use Laravel helpers over raw PHP
  - Prefer Eloquent over raw queries
  - Form Requests for validation
  - API Resources for API responses

Vue 3 Standards:
  - Always <script setup> syntax
  - JSDoc or defineProps generic for TypeScript types
  - Small, focused components
  - @ alias for imports

Naming Conventions:
  - Controllers: UserController, LeadController
  - Models: User, Lead, Conversation
  - Migrations: create_users_table, add_score_to_leads_table
  - Vue Pages: resources/js/Pages/Client/Dashboard.vue
  - Vue Components: resources/js/Components/LeadCard.vue

Security:
  - HTTPS only in production
  - Never log sensitive data (passwords, API keys, tokens)
  - Hash::make() for passwords
  - Validate all user input
  - Escape output in Blade
  - CSRF protection on all non-API forms

Database Conventions:
  - Plural snake_case table names
  - Always include id, created_at, updated_at
  - Soft deletes where appropriate (deleted_at)
  - Tenant-scoped tables MUST have tenant_id
  - Foreign keys: {singular_table}_id
  - Always index foreign keys
  - Cascading deletes: use cautiously

Git Commit Format:
  - type(scope): description
  - Types: feat, fix, refactor, test, docs, chore
  - Co-Authored-By: Claude <noreply@anthropic.com>

## TOPIC: Development Workflow

source: CLAUDE.md

Feature Development Process (non-trivial features):
  1. Brainstorm (superpowers:brainstorming) — lock decisions before writing spec
  2. Spec → docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
  3. Plan → docs/superpowers/plans/YYYY-MM-DD-<topic>.md (TDD-shaped tasks)
  4. Self-review plan before executing
  5. Task 0 for unverified assumptions (external API, env config, third-party behavior)
  6. Branch + subagent-driven execution
  7. Three test layers: TDD per task, full suite between tasks, E2E browser smoke before PR
  8. Pint → /simplify → Pint → /simplify → PR
  9. PR description per template (Summary, Deploy steps, behavior changes, Test plan, links)
  10. After merge: update memory

Testing Layers:
  - Layer 1: TDD (failing test → implementation → green)
  - Layer 2: Full suite between tasks (php artisan test — not feature-scoped)
  - Layer 3: E2E browser smoke before PR (Playwright or manual)

Pint/Simplify Order:
  - Pint first (deterministic, cheap)
  - /simplify (three parallel reviewers: reuse/quality/efficiency)
  - Pint again (catches simplify-introduced style issues)
  - /simplify again (catches simplify-introduced logic issues)

Debugging:
  - Specific console.log with labels — never "check the console"
  - (IS $) prefix for billed external calls, (NO $) for local/free calls
  - Tell user exact log label and exact properties needed

Legacy Code:
  - Always delete legacy code — no dual-system support
  - Remove backward compatibility completely when migrating
  - No fallback logic (oldPattern ?? newPattern) — choose one

## TOPIC: Multi-Tenancy Architecture

source: CLAUDE.md, docs/superpowers/specs/2026-05-14-architecture-deepening-design.md

Pattern: Single database with tenant_id column on all tenant-scoped tables
Framework: Spatie Laravel Multitenancy
Auth: tenant-aware middleware on all client routes; admin routes NOT tenant-scoped

Route Structure:
  / → Marketing/landing (public)
  /login, /register → Auth (public)
  /dashboard/* → Client dashboard (tenant-scoped)
  /admin/* → Admin dashboard (platform admin only)

  /api/v1/* → Client API (Sanctum auth)
  /api/admin/* → Admin API

Key Directory Structure:
  app/Http/Controllers/Admin/ — admin dashboard
  app/Http/Controllers/Api/V1/Widget/ — public widget API

## TOPIC: Cache Invalidation Ownership

source: memory/tenant_saved_cache.md (referenced from architecture memory)

Cache invalidation for tenant:{id}:with_plan key is owned by the Tenant::saved hook.
Do NOT add explicit Cache::forget calls for this key elsewhere.
BustsTenantUsageCache trait owns invalidation for tenant usage cache (fires on model `created`).
These are single sources of truth — duplicating them creates confusing behavior in tests and production.
