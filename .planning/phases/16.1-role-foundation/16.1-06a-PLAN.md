<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server share() payload → browser | Crossed once per Inertia request; UI booleans are ADVISORY — the backend `$this->authorize()` (Plans 04/05) is canonical |
| Vue template `$page.props.auth.user.can` → DOM render | If can map is stale (caching bug), a tenant Manager could see Owner-only buttons. Acceptance: share() runs PER REQUEST, no caching |
| Inertia Link form post (chooser) → ChooseRoleController | Crossed by dual-role users picking a context; defended in Plan 03 by per-context role check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16.1.06a-01 | Information Disclosure | Vue can map exposes ability names | accept | The keys are stable URL slugs already visible in routes (e.g. `/dashboard/billing` reveals "billing"); knowing the ability key adds no actionable surface for a tenant agent. Backend `$this->authorize()` is the gate |
| T-16.1.06a-02 | Tampering | client modifies `can.manage_billing = true` in dev tools | mitigate | UI booleans are ADVISORY — every mutation hits a backend controller that ALSO authorizes (Plan 05). Local tampering surfaces "button visible but click 403s" rather than data damage |
| T-16.1.06a-03 | Elevation of Privilege | stale `auth.user.can` map after role revocation | mitigate | share() resolves the can map PER REQUEST via Gate::forUser — no caching. After a role revocation, the next page load reflects the new state. The buildAbilityMap method is NOT memoized in any singleton. Acceptance criterion (Task 1): N+1 protection via loadMissing, NOT cross-request memoization |
