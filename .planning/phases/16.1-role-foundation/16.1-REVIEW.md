# Phase 16.1: Code Review Report

**Reviewed:** 2026-05-21T05:30:00Z
**Depth:** standard
**Files Reviewed:** 33
**Status:** issues_found

## Summary

Phase 16.1 replaces the legacy `admin_users` table and `role` column on `users` with a proper `user_roles` pivot, collapses to a single `web` guard, introduces 13 named Gates backed by `RolePermissions`, and sweeps all Vue pages for `v-if` gates on mutation CTAs. The implementation is structurally sound — enum-backed roles, hierarchical `hasRoleAtLeast()`, policy stacking, and Inertia shared `auth.user.can` map all read correctly.

**Webhook routes verified:** Stripe/Cashier and DK Bank callback routes are confirmed outside the `require.super_admin` middleware group in `routes/web.php`. Focus area 3 is clean.

Four areas require attention before ship:

1. **Widget API key leaked to Agents** — `WidgetController::index()` passes the plaintext `api_key` in the Inertia props payload with no role gate. Any authenticated tenant user (including Agent tier) can read the widget API key from the page props, regardless of whether the UI exposes it.

2. **MySQL NULL unique index** — the `user_roles` composite unique index does not prevent duplicate SuperAdmin rows (MySQL treats each NULL as distinct). The application-layer `hasRole()` logic still works, but the DB-level guarantee that enforces D-09 is absent.

3. **`assert()` in production-critical paths** — `Controller::getTenant()` and `Controller::getUser()` use `assert()` to enforce non-null invariants. With `zend.assertions=0` (common in production PHP tuning), both assertions are compile-time stripped. A super_admin user who hits a client route gets a `TypeError` at an unpredictable call site rather than a clean 403.

4. **Intended-URL loss for dual-role login** — `LoginController::store()` redirects dual-role users to `route('login.choose')` without `redirect()->intended()`, discarding any pre-login destination. Both single-role branches correctly use `redirect()->intended()`.

## Critical Issues

### CR-01: `WidgetController::index` exposes plaintext `api_key` to all authenticated tenant users

**File:** `app/Http/Controllers/Client/WidgetController.php:25`

**Issue:** `WidgetController::index()` passes `'api_key' => $tenant->api_key` in the Inertia response with no authorization check. `api_key` is the widget secret used to authenticate all public widget API calls — it controls which domain can embed the chatbot and who can initiate conversations. Any authenticated user with a tenant (Owner, Manager, or Agent) receives it in the page props payload on every visit to the Widget Settings page.

The `update()` and `regenerateApiKey()` actions are correctly gated behind `Ability::ManageTenantSettings` (Owner-only). But the read path for `index()` has no gate at all. An Agent-tier user who navigates to `/widget-settings` (or reads `window.__INERTIA_PROPS__` in browser devtools) gets the full key.

`api_key` is a tenant credential. Agents should not have it. The CLAUDE.md security requirement "Never log sensitive data (passwords, API keys, tokens)" applies equally to response payloads.

**Fix — Option A (recommended):** Gate `index()` at the same level as the mutation actions:

```php
public function index(Request $request): Response
{
    $this->authorize(Ability::ManageTenantSettings->value);  // Owner+ only
    $tenant = $this->getTenant($request);
    // ... rest unchanged
}
```

**Fix — Option B:** If Agents legitimately need the Widget Settings page for some fields (e.g., embed URL, bot appearance), omit `api_key` from the props and add a separate `show-api-key` action gated behind `ManageTenantSettings`:

```php
'tenant' => [
    'id' => $tenant->id,
    'name' => $tenant->name,
    // Remove: 'api_key' => $tenant->api_key,
    'api_key' => Gate::allows(Ability::ManageTenantSettings->value)
        ? $tenant->api_key
        : null,
    'settings' => $tenant->settings ?? [],
],
```

### CR-02: MySQL NULL unique index does not enforce SuperAdmin row uniqueness

**File:** `database/migrations/2026_05_20_000008_create_user_roles_table.php:21`

**Issue:** The migration creates `unique(['user_id', 'role', 'tenant_id'])` on `user_roles`. MySQL's B-tree unique index treats each NULL value as distinct from all other NULLs. For SuperAdmin rows, `tenant_id` is NULL. The DB will silently accept a second (or third) `(user_id, 'super_admin', NULL)` row — the unique constraint does not fire.

`hasRole()` on the User model iterates the in-memory collection, so duplicates do not cause incorrect `true/false` results today. However:

- Running `php artisan db:seed --class=DatabaseSeeder` a second time without a prior rollback silently inserts duplicate rows, because `Tenant::create` and `UserRole::create` have no existence check.
- `UserRole::firstOrCreate(['user_id'=>$id, 'role'=>Role::SuperAdmin, 'tenant_id'=>null])` has a TOCTOU window where two concurrent requests can both "first" and both "create" — the DB unique constraint was the only hard stop.
- Future code that calls `->sole()` or asserts `->count() === 1` on the pivot will break.
- Decision D-09 ("super_admin MUST have tenant_id=NULL, enforced by schema") is stated as a schema guarantee but is not actually enforced at the DB level.

**Fix — Option A (correct, requires new migration):** Use a virtual/generated column to coerce NULL to 0, which is indexable for uniqueness:

```php
Schema::create('user_roles', function (Blueprint $table) {
    $table->id();
    $table->foreignId('user_id')->constrained()->cascadeOnDelete();
    $table->enum('role', ['super_admin', 'owner', 'manager', 'agent']);
    $table->foreignId('tenant_id')->nullable()->constrained()->cascadeOnDelete();
    $table->unsignedBigInteger('tenant_id_coalesced')
        ->virtualAs('COALESCE(tenant_id, 0)');
    $table->timestamps();
    $table->index(['user_id', 'tenant_id']);
    $table->index(['tenant_id', 'role']);
    $table->unique(['user_id', 'role', 'tenant_id_coalesced'],
        'user_roles_user_role_tenant_unique');
});
```

**Fix — Option B (application-layer, no schema change):** Add a `creating` hook in `UserRole::booted()`:

```php
static::creating(function (UserRole $ur): void {
    $exists = static::where('user_id', $ur->user_id)
        ->where('role', $ur->role)
        ->where('tenant_id', $ur->tenant_id)  // Eloquent generates IS NULL for null
        ->exists();
    if ($exists) {
        throw new \RuntimeException(
            "Duplicate UserRole: user {$ur->user_id} already holds role {$ur->role->value}"
        );
    }
});
```

Note: Eloquent `->where('tenant_id', null)` correctly generates `WHERE tenant_id IS NULL`.

## Warnings

### WR-01: `assert()` used for production-critical null guards in Controller base class

**File:** `app/Http/Controllers/Controller.php:24,39`

**Issue:** Both `getUser()` (line 24) and `getTenant()` (line 39) use `assert()` to enforce that `Auth::user()` is a non-null `User` and that `$user->tenant` is a non-null `Tenant`. In production PHP deployments it is common to set `zend.assertions=-1` (compile-time strip) or `zend.assertions=0` (runtime no-op). When stripped, both methods silently return the wrong type (`null` coerced as `User`/`Tenant` by the declared return type), and the first property access on the returned value throws a `TypeError` at an arbitrary downstream call site with no indication of the real problem.

`getTenant()` is specifically dangerous for the dual-role case: a SuperAdmin user who bookmarks `/dashboard` and visits it after choosing the admin context has `$user->tenant === null`. With assertions enabled this throws `AssertionError` immediately (helpful). With assertions disabled it returns null typed as Tenant, and the crash site is wherever the caller first accesses a property — not in `getTenant()`.

**Fix:** Replace both `assert()` calls with explicit conditional aborts that are always enforced:

```php
protected function getUser(): User
{
    $user = Auth::user();
    if (! $user instanceof User) {
        abort(401);
    }
    return $user;
}

protected function getTenant(?Request $request = null): Tenant
{
    $user = $request?->user() ?? Auth::user();
    if (! $user instanceof User) {
        abort(401);
    }
    $tenant = $user->tenant;
    if (! $tenant instanceof Tenant) {
        abort(403, 'No tenant context for this user.');
    }
    return $tenant;
}
```

### WR-02: Dual-role login loses intended redirect destination

**File:** `app/Http/Controllers/Auth/LoginController.php:56`

**Issue:** When a user holding both SuperAdmin and a tenant role logs in, they are redirected to `route('login.choose')` via a plain `redirect()->route(...)`. This discards the pre-login URL that Laravel stashed in the session via `redirect()->guest()`. The two single-role branches on lines 57–58 correctly use `redirect()->intended()`. The dual-role branch does not.

A dual-role user (e.g., `test@example.com`) who tries to visit `/dashboard/billing` while unauthenticated will be sent to the chooser after login and then to `/dashboard` — the intended URL is silently dropped.

**Fix:** Propagate the intended URL through the chooser flow. In `LoginController::store()`:

```php
$isSuperAdmin && $hasTenantRole => redirect()->route('login.choose')
    ->with('url.intended', session()->pull('url.intended', '')),
```

Then in `ChooseRoleController::store()` consume it:

```php
// context === 'tenant':
return redirect()->to(session()->pull('url.intended', route('dashboard')));

// context === 'admin':
return redirect()->to(session()->pull('url.intended', route('admin.dashboard')));
```

### WR-05: `ChooseRoleController` calls `$request->user()` without null check

**File:** `app/Http/Controllers/Auth/ChooseRoleController.php:24,70`

**Issue:** Both `create()` (line 24) and `store()` (line 70) call `$request->user()` and immediately chain `->loadMissing('roles')` without null checking. The `auth` middleware upstream should guarantee a non-null user — but unlike `LoginController`, which uses a `@var User $user` docblock after `$request->authenticate()` to document the invariant, `ChooseRoleController` has no annotation and no guard. A misconfiguration that bypasses `auth` middleware causes a `TypeError` with no actionable message.

**Fix:** Add a null guard matching the `auth` contract:

```php
// In create() line 24 and store() line 70:
/** @var \App\Models\User $user */
$user = $request->user();
if ($user === null) {
    return redirect()->route('login');
}
```

Alternatively, use the inherited `getUser()` from the Controller base class (which already has the null-check pattern after WR-01 is fixed).

### WR-06: `BillingController` read pages lack authorization; any tenant user can view billing info

**File:** `app/Http/Controllers/Client/BillingController.php` (`index`, `plans`, `transactions` methods)

**Issue:** `BillingController::index()`, `plans()`, and `transactions()` have no `$this->authorize()` calls. Any authenticated tenant user — including Agent tier — can browse `/dashboard/billing`, `/dashboard/billing/plans`, and `/dashboard/billing/transactions`. The mutation actions (`submitPayment`, `activateTrial`) are correctly gated behind `ManageBilling` (Owner-only). The read-only pages are not gated at all.

Whether this is intentional (billing transparency for all roles) or an oversight depends on product intent. If the decision is that Agents should not see billing information, each read method needs a gate. If billing read access is acceptable for any role, the intent should be documented in a comment.

**Fix (if read access should be Owner-only):**

```php
public function index(): Response
{
    $this->authorize(Ability::ManageBilling->value);
    // ... rest unchanged
}
```

**Fix (if read access is intentionally open to all roles):** Add an explicit comment:

```php
// Billing read pages are intentionally accessible to all tenant roles (no authorization call).
// Only mutation endpoints (submitPayment, activateTrial) require ManageBilling.
public function index(): Response
```

### IN-02: `RoleBadge.vue` diverges from PATTERNS.md spec by not using `<Badge>` component

**File:** `resources/js/Components/RoleBadge.vue`

**Issue:** The component renders a custom styled `<span>` with hardcoded Tailwind classes instead of composing `<Badge variant="...">` from `@/Components/ui/badge`. PATTERNS.md called for using the Badge component. This is not a functional defect but creates divergence requiring parallel maintenance when Badge variants change.

**Fix:** Replace with a `<Badge>` composition:

```vue
<script setup>
import { computed } from 'vue'
import { Badge } from '@/Components/ui/badge'

const props = defineProps({ role: { type: Object, default: null } })

const variantMap = {
  super_admin: 'destructive',
  owner: 'default',
  manager: 'secondary',
  agent: 'outline',
}
const variant = computed(() => variantMap[props.role?.value] ?? 'outline')
</script>

<template>
  <Badge v-if="role" :variant="variant">{{ role.label }}</Badge>
</template>
```

### IN-03: `tests/bootstrap.php` hardcodes an absolute machine-specific path

**File:** `tests/bootstrap.php:12`

**Issue:** Line 12 hardcodes `/Users/sam/Dev/laravel/chatbot/vendor` as `$mainVendor`. This path is machine-specific. CI runners, teammates, or a second local clone at a different path will get a fatal `require` failure, making the entire test suite unrunnable on any machine other than the original author's. The worktree root (`$worktreeRoot`) is correctly derived dynamically — only the vendor path is hardcoded.

**Fix:** Derive the vendor path dynamically, with an env-var override for the worktree-sharing use case:

```php
$worktreeRoot = dirname(__DIR__);

// Use the worktree's own vendor by default; SHARED_VENDOR_PATH allows worktree sharing.
$mainVendor = getenv('SHARED_VENDOR_PATH') ?: $worktreeRoot . '/vendor';
```

Document the env var in `phpunit.xml` or `.env.example` so the worktree-sharing pattern is discoverable without reading bootstrap source.
