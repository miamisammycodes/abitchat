# Phase 16.1: Role Foundation - Context

**Gathered:** 2026-05-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 16.1 delivers a unified four-role RBAC foundation for the entire system. It collapses the existing two-table / two-guard split (`users` for tenant users + `admin_users` for platform admins) into a single `users` table backed by a multi-role pivot, so one person can simultaneously be a platform `super_admin` and the `owner` of their own tenant.

Concretely, 16.1 ships:

1. **Backed PHP enum `App\Enums\Role`** with ordinal rank: `SuperAdmin > Owner > Manager > Agent`. Hierarchical: `Owner` includes `Manager` includes `Agent` for ability checks within a tenant. `SuperAdmin` lives outside the tenant hierarchy entirely.
2. **`user_roles` pivot table** — `(id, user_id, role, tenant_id NULL, timestamps)`. One user can hold multiple roles (`super_admin` + `owner` is the headline case).
3. **Schema cleanup** — drop `admin_users` table, delete `AdminUser` model, drop the legacy `users.role` enum column. `users.tenant_id` becomes nullable (`NULL` for `super_admin` rows; non-null for tenant users; tenant-scoped roles in `user_roles` must match `users.tenant_id`).
4. **Single auth guard** — remove the `admin` guard; everyone authenticates via `web`. `/admin/*` is gated by a new `RequireSuperAdmin` middleware. Single `/login` page; post-login redirect is role-aware (super_admin → /admin/dashboard; tenant user → /dashboard; dual-role → small chooser).
5. **`App\Auth\RolePermissions`** — canonical const map of ability → min role, exhaustively tested.
6. **Laravel Gates** — every ability registered via `Gate::define()` in `AuthServiceProvider`, backed by `RolePermissions::can()`. Existing Eloquent policies (`ConversationPolicy`, `LeadPolicy`, `KnowledgeItemPolicy`, `TransactionPolicy`) stack tenant-ownership + role-tier (e.g., `LeadPolicy::view` requires `Agent+`, `LeadPolicy::delete` requires `Manager+`).
7. **Full UI sweep** — `HandleInertiaRequests` shares a resolved `auth.user.can = { manage_billing: bool, ... }` map. Every `Pages/Client/*` and `Pages/Admin/*` surface gains `v-if="$page.props.auth.user.can.X"` for owner-only / manager+ / agent+ controls. Top-nav role badge next to user name; dual-role users see a context-switch chip on `/admin`.
8. **Fix the trigger** — `UpdateWebsiteIndexingRequest::authorize()` migrates from `isOwner()` to `Gate::allows('manage-tenant-settings')`. `DatabaseSeeder` rebuilt to seed the full role matrix so the 403 disappears.
9. **Tests** — role × ability matrix coverage; PHPStan baseline = 0 maintained.

Phase 17 (Team Management — invites, SSO, ownership transfer, multi-tenant user membership, conversation assignment) is the layer that sits ON TOP of this foundation. 16.1 explicitly does NOT ship invites, SSO, or multi-tenant membership.

</domain>

<decisions>
## Implementation Decisions

### Role Taxonomy
- **D-01: Four roles total** — `super_admin`, `owner`, `manager`, `agent`. `super_admin` is the platform-level role; `owner`/`manager`/`agent` are tenant-scoped roles. Phase 17 ROADMAP wording ("owner, editor, viewer") is superseded by this decision; the planner should update Phase 17 SC2 wording during planning of Phase 17.
- **D-02: Hierarchical within tenant** — `Owner ≥ Manager ≥ Agent`. `Owner` implicitly satisfies `Manager+` and `Agent+` checks. `SuperAdmin` is NOT part of the tenant hierarchy — it is a separate, orthogonal role.
- **D-03: Multi-role storage** — one user can hold multiple roles simultaneously (the canonical dual-role case is `super_admin` + `owner`-of-test-workspace). Storage = `user_roles(user_id, role, tenant_id NULL)` pivot table. Drop the legacy `users.role` enum column.
- **D-04: Backed PHP enum** — replace the existing `User::isOwner()`/`isManager()`/`isAgent()` helpers with `App\Enums\Role` (backed by string values matching the pivot column) + `User::hasRole(Role, ?Tenant = current)` and `User::hasRoleAtLeast(Role, ?Tenant = current)`. The 4 existing helpers in `app/Models/User.php:54–72` are deleted in this phase.

### Architecture (Unified RBAC)
- **D-05: Drop the `admin_users` table and `admin` guard entirely** — no production data to preserve (confirmed by user: "since there is no prod for this system we can do everything from scratch"). The `AdminUser` model is deleted. Existing admin routes stay at `/admin/*` and are protected by a new `App\Http\Middleware\RequireSuperAdmin` middleware.
- **D-06: `users.tenant_id` stays, becomes nullable** — `super_admin`-only rows have `tenant_id = NULL`; tenant users keep `tenant_id = X`. Tenant users are still 1×1 with a tenant for 16.1 (one user, one tenant). For any non-`super_admin` row in `user_roles`, `user_roles.tenant_id` MUST equal `users.tenant_id` (enforced application-side; planner decides whether to also add a DB-level CHECK). Multi-tenant-per-user is Phase 17 territory.
- **D-07: BelongsToTenant + DEC-05 (NoRawTenantIdWhere Larastan rule) keep working unchanged** — tenant users still have `tenant_id` set; `Model::forTenant($tenant)` is still canonical. The User model continues using `BelongsToTenant`.
- **D-08: Single `/login` page; role-aware post-login redirect** — successful login: if user holds only `super_admin` → `/admin/dashboard`; only tenant role(s) → `/dashboard`; both → render a tiny chooser ("Continue as Platform Admin" / "Continue as Owner of [Tenant]"). Drop `/admin/login` route + view.

### Permission Matrix (canonical, shipped in 16.1)
The full matrix lives in `App\Auth\RolePermissions` as const arrays + a unit test asserting the map. Initial matrix:

**SuperAdmin only** (no tenant context required; ability checks ignore tenant):
- `view-admin-dashboard` — covers entire `/admin/*` tree
- `manage-tenant-as-admin` (the existing `Admin\ClientController` editing of tenant fields, including `bot_custom_instructions` — bot brain stays admin-only per **D-09**)
- `manage-platform-settings` (placeholder for future)

**Owner only** (tenant role, exact match):
- `manage-billing` (Stripe + DK Bank payment flows, plan changes, transactions, invoices)
- `manage-team` (Phase 17 surface — invite, remove, change role, transfer ownership; ability defined now, enforced when Phase 17 ships UI)
- `manage-tenant-settings` (website URL + auto_recrawl, widget brand: welcome_message, primary_color, position)
- `delete-tenant` (Phase 22 surface — defined now)

**Manager+** (Owner OR Manager):
- `manage-knowledge-base` (KB content CRUD only — upload, edit, delete, re-embed, attach. NOT bot brain.)
- `manage-integrations` (future; placeholder)
- `view-analytics-full` (full analytics dashboard)

**Agent+** (Owner OR Manager OR Agent):
- `manage-conversations` (view, take-over, edit, delete)
- `manage-leads` (view, edit, delete, export, merge — when shipped)
- `view-dashboard` (basic landing dashboard)

- **D-09: Bot brain is platform-admin only and stays outside the tenant role hierarchy entirely.** `tenants.bot_custom_instructions` is editable only via `Admin\ClientController` / `Admin/Clients/Show.vue`. NO tenant user (not even `owner`) can edit it. This matches the current codebase (no tenant-side route exists). The planner should add a guard test asserting this remains true even if a new tenant-side route is accidentally added later.

### Enforcement Pattern
- **D-10: Laravel Gates everywhere** — `Gate::define('manage-billing', fn(User $u) => App\Auth\RolePermissions::can($u, Ability::ManageBilling, $u->tenant))` in `AuthServiceProvider`. Call sites use `$this->authorize('manage-billing')` in controllers and FormRequest `authorize()`. Vue receives the resolved boolean via `auth.user.can.manage_billing`.
- **D-10 (correction 2026-05-20):** Per PATTERNS.md fact #1, this project has no `AuthServiceProvider`. Gates register in `app/Providers/AppServiceProvider::boot()` alongside the existing `RateLimiter::for(...)` blocks. Plan 16.1-04 enacts this. (Audit-trail correction — original D-10 wording preserved above.)
- **D-11: Existing Eloquent policies are upgraded to stack tenant-ownership + role-tier** — `LeadPolicy::view` still checks `$lead->tenant_id === $user->tenant_id` AND additionally checks `Agent+`. `LeadPolicy::delete` checks ownership AND `Manager+`. Same pattern for `ConversationPolicy`, `KnowledgeItemPolicy`, `TransactionPolicy`. All four policies delegate the role-tier check to `RolePermissions::can($user, Ability::X)` so the matrix has one source of truth.
- **D-12: Optional but recommended — typed `Ability` enum.** `App\Enums\Ability` (backed by string) eliminates string typos at call sites and is PHPStan-friendly (DEC-09 baseline = 0). Decision: ship it. Gate names = `Ability::ManageBilling->value`.

### UI Exposure
- **D-13: Resolved abilities shared via Inertia** — `HandleInertiaRequests::share()` resolves every registered ability via `Gate::check()` once per request and exposes a flat map `auth.user.can = { manage_billing: bool, manage_team: bool, ... }`. No role-checking logic in Vue — frontend reads booleans only. Renames of role mappings don't touch any `.vue` file.
- **D-14: Full UI sweep in 16.1** — every `Pages/Client/*.vue` and `Pages/Admin/*.vue` surface that exposes an owner-only / manager+ / agent+ control gets a `v-if="$page.props.auth.user.can.X"`. Reconciles with the earlier "tiered ship" answer: since the UI gate is a thin reflection of the backend gate, both ship together in 16.1.
- **D-15: Top-nav role badge + dual-role context chip** — small role badge ("Owner" / "Manager" / "Agent" / "Platform Admin") next to user avatar in the existing `AvatarMenu` component. On `/admin/*` pages, dual-role users see an extra chip ("Switch to my workspace [Tenant Name]") that links to `/dashboard`. Symmetric chip on `/dashboard/*` for super_admin+tenant users ("Switch to admin dashboard").

### Seeder & Test Setup
- **D-16: DatabaseSeeder rebuilt to seed the full matrix.** Accounts:
  - `admin@example.com` — `super_admin` only, `tenant_id NULL`, password `password`
  - `test@example.com` — `owner` of "Test Company" + `super_admin` (dual-role to exercise the chooser flow), password `password`
  - `manager@example.com` — `manager` of "Test Company", password `password`
  - `agent@example.com` — `agent` of "Test Company", password `password`
  - "Demo Co" tenant with its own owner@demo.example / manager@demo.example / agent@demo.example
  All Plans + the existing test Tenant fields stay the same.
- **D-17: Update CLAUDE.md test credentials** — current "Client User" / "Admin User" entries are replaced with the new four-account matrix above so future sessions know who to log in as for each role.
- **D-18: RegisterController updated to write a user_roles row** — public signup creates a User (with `tenant_id` set) + inserts a `user_roles(user_id, 'owner', tenant_id)` pivot row inside the same DB transaction. No self-registration path for `super_admin` (must be seeded or created via tinker).

### Testing
- **D-19: Role × ability matrix tests** — for each ability, test that the minimum required role passes and lower roles fail (Pest feature tests against representative controllers/routes).
- **D-20: Browser smoke test** — Playwright or manual: log in as each of the four seeded accounts and confirm the dashboard surfaces match expectations (buttons visible/hidden, /admin gated, dual-role chooser appears for `test@example.com`).
- **D-21: PHPStan baseline stays at 0** — typed enums for both `Role` and `Ability`, fully-typed Gate closures, no string-keyed magic.

### Claude's Discretion
- Whether `user_roles.tenant_id` constraint is enforced via DB CHECK or app-level only (planner's call based on MySQL version + portability cost).
- Exact column ordering and index choices on `user_roles` (planner picks; index on `(user_id, role)` and `(tenant_id, role)` are the obvious candidates).
- Migration ordering and rollback strategy (planner specifies; user already approved "rebuild from scratch", but the migration should still be idempotent for `migrate:fresh --seed`).
- Whether the dual-role chooser is a full intermediate page or an inline modal (UI researcher / Phase 16.1 design discretion if `/gsd:ui-phase` is invoked).
- Exact role-badge styling and chip placement (UI researcher's call).
- Whether Phase 17's planned UI for invites should be sketched at all in 16.1 (preferred: no — keep 16.1 strictly foundation).

<code_context>
## Existing Code Insights

### Reusable Assets
- **`BelongsToTenant` trait** (`app/Models/Concerns/BelongsToTenant.php`) — User already uses it; the pivot model `App\Models\UserRole` should also use it for any tenant-scoped queries. Provides `forTenant($tenant)` + the Larastan-enforced canonical query API.
- **Existing Eloquent policies** (`app/Policies/{Conversation,Lead,KnowledgeItem,Transaction}Policy.php`) — tenant-ownership checks are already in place. Layering role-tier on top is additive, not a rewrite.
- **Existing `$this->authorize()` call sites** — already present in `BillingController`, `DkBankQrController`, `KnowledgeBaseController`, `LeadController`. The new Gate-based ability checks (`$this->authorize('manage-billing')`) slot in naturally next to the existing model-policy authorize calls.
- **`AvatarMenu` component** (likely in `resources/js/Components/`) — top-nav slot already exists for user info; role badge + context chip plug in there.

### Established Patterns
- **Spec corrections in PROJECT.md** — `bot_custom_instructions` max 1000 chars (PR #11 prompt injection defense). New ability matrix MUST NOT allow tenant users to edit this field, regardless of role.
- **Strict typing everywhere** — `declare(strict_types=1)`, typed return types, typed parameters. Apply to new enums, RolePermissions, middleware, Gate closures.
- **PHPStan level-max baseline = 0** — every new file passes Larastan with no exclusions. Backed enums for both Role and Ability are the path of least friction.
- **`BustsTenantUsageCache` trait pattern** — model observers that bust caches on `created`. Not directly applicable here, but the User model already participates in tenant-scoped caching; verify `Tenant::saved`-style cache invalidation doesn't break when a User has multiple `user_roles` rows.
- **Existing admin-tenant editing in `Admin\ClientController:204`** — `bot_custom_instructions` validation lives there; verify no tenant-side validation creeps in during the controller sweep.
- **Inertia share** (`HandleInertiaRequests.php`) — already shares `auth.user`, `auth.admin`, and `tenant` separately. After the merge, the structure flattens: `auth.user` (with embedded `roles[]` and `can{}` map) + `tenant` (unchanged); `auth.admin` is removed.

### Integration Points
- **Where the new code connects to existing system:**
  - `AuthServiceProvider::boot()` — register every Gate with `Gate::define('ability-name', closure)`.
  - `HandleInertiaRequests::share()` — replace `auth.admin` separately-shared block with a unified `auth.user.can` map derived from `RolePermissions::all()`.
  - `RouteServiceProvider` (or `routes/admin.php`) — `Route::middleware(['auth', 'role:super_admin'])->prefix('admin')` replaces the current `auth:admin` middleware group.
  - `LoginController::authenticated()` / `LoginResponse` — implement the role-aware redirect.
  - `User::tenant` relationship — stays (1×1 belongs-to). New `User::roles()` HasMany relationship to `UserRole` pivot.

### Risk Surface (planner should plan tests for)
- **Auth middleware ordering** — `auth` → `RequireSuperAdmin` → `RequireWebsiteIndexing` etc. Off-by-one in middleware ordering caused PR #15's TrustProxies bug; learn from that.
- **Session/cookie regeneration** — if anyone is currently logged in with a session cookie when the merge ships, sessions become invalid (admin guard gone). No prod = safe; verify local dev re-login works after `migrate:fresh --seed`.
- **Race condition on user_roles inserts** — RegisterController must insert in the same DB transaction as User::create + Tenant::create (existing transaction already wraps these).
- **Octane / SessionTokenService-style singleton bugs** — see PR #30 CONS-22-d lesson. Any Gate-related caching must be request-scoped, not Octane worker-scoped.
- **Stripe / Cashier webhook routes** — webhook handlers are NOT user-authenticated; verify they don't accidentally fall under `RequireSuperAdmin` (they should be in a separate route group with signature verification).
- **DK Bank QR callback routes** — same concern: callback URLs are anonymous; verify route placement is outside the auth middleware tree.

## Specific Ideas
