# Phase 16.1 — UI Design Contract

> Visual and interaction contract for the role-aware UI additions introduced by Phase 16.1 (Role Foundation).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** This contract covers ONLY the five net-new role-aware surfaces:
> 1. Dual-role chooser page (`/login/choose`)
> 2. Top-nav role badge in `AdminLayout` + `ClientLayout`
> 3. Dual-role context-switch chip in the top nav
> 4. `v-if` visual rules for role-gated controls across `Pages/Client/*` + `Pages/Admin/*`
> 5. Single `/login` page (existing form retained; redirect logic only)
>
> Existing dashboards, KB, billing, leads, conversations remain visually unchanged except for `v-if` wrappers on role-gated controls.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | manual (shadcn-style hand-rolled primitives; no `components.json`, no CLI) |
| Preset | not applicable (Vue 3 + Inertia stack — shadcn CLI is React-only) |
| Component library | radix-vue + class-variance-authority + tailwind-merge (already installed; see `package.json`) |
| Icon library | `lucide-vue-next` ^1.0 (locked — already a direct dependency of this project; verified in `package.json`) |
| Font | Inter (defined in `resources/css/app.css` `--font-sans`) |
| CSS framework | Tailwind v4 with shadcn HSL CSS-variable theme (locked in `resources/css/app.css`) |

**Anti-patterns forbidden by this contract:**
- Introducing a new icon library (e.g., Heroicons, Phosphor). Use `lucide-vue-next` only.
- Adding new color tokens beyond what is declared in `app.css`. No literal HSL or hex in this phase's Vue templates — all colors flow through the existing `--*`/`--*-foreground` variables via Tailwind utility classes (`bg-primary`, `text-destructive`, etc.).
- Using `destructive` variant on the role badge. `destructive` is reserved exclusively for destructive actions (see Color table). Platform Admin uses `outline + Shield icon + text-foreground` — discriminable from Agent via the icon, not color.
- Showing the literal string "Super Admin" anywhere user-facing. The UI label is "Platform Admin" (per `<specifics>` in CONTEXT.md).
- Storing the dual-role active context in a session key. The URL prefix (`/admin/*` vs `/dashboard/*`) IS the active context — no `session('active_context')` indirection.
- Hiding owner-only controls via `disabled`/`opacity-50`. D-14 mandates `v-if` — buttons collapse out of the DOM, layout reflows.

| Role | Value | Usage in this phase |
|------|-------|---------------------|
| Dominant (60%) | `--background` = `0 0% 100%` (light) / `222.2 84% 4.9%` (dark) | Chooser page background; login page background; layout body background |
| Secondary (30%) | `--card` (`0 0% 100%` / `222.2 84% 4.9%`) + `--muted` (`210 40% 96.1%`) | Card surfaces on chooser; sidebar; top nav band; ContextSwitchChip background (`bg-muted`) |
| Accent (10%) | `--primary` = `221.2 83.2% 53.3%` (light) / `217.2 91.2% 59.8%` (dark) | Chooser "Continue as Platform Admin" + "Continue as Owner" buttons; Owner role badge; active sidebar item (pre-existing); focus rings; primary CTAs only |
| Destructive | `--destructive` = `0 84.2% 60.2%` (light) / `0 62.8% 30.6%` (dark) | Destructive actions only (logout text — pre-existing; usage-limit critical banner — pre-existing). **NOT used by any role badge.** |

**Accent reserved for:** chooser primary action buttons, Owner role badge background, primary CTA buttons across the layouts, active sidebar nav item, focus-visible rings, link hover states inside Card components. Never used for "every interactive element."

### Role Badge Color Mapping (this phase's load-bearing color decision)

Four distinct, accessible mappings. **Color alone does NOT discriminate** — every role badge carries a leading icon, so colorblind users can distinguish via icon + label, and screen reader users via label text alone.

| Role enum (`App\Enums\Role`) | UI label | Badge variant | Lucide icon | WCAG AA on `bg-card` |
|------------------------------|----------|--------------|-------------|----------------------|
| `Role::SuperAdmin` | `Platform Admin` | `outline` (transparent bg, `text-foreground` border-input) | `Shield` (h-3.5 w-3.5, mr-1) | foreground on card: ≥ 15:1 (passes) |
| `Role::Owner` | `Owner` | `default` (`bg-primary text-primary-foreground`) | `Crown` (h-3.5 w-3.5, mr-1) | primary-foreground on primary: 8.6:1 (passes AAA) |
| `Role::Manager` | `Manager` | `secondary` (`bg-secondary text-secondary-foreground`) | `Users` (h-3.5 w-3.5, mr-1) | secondary-foreground on secondary: ≥ 15:1 (passes) |
| `Role::Agent` | `Agent` | `outline` (transparent bg, `text-foreground` border-input) | `Headphones` (h-3.5 w-3.5, mr-1) | foreground on card: ≥ 15:1 (passes) |

**Why Platform Admin and Agent share `outline` variant:** Color alone would force a fifth tone outside the locked `app.css` palette. The icon (`Shield` vs `Headphones`) + label text carry the full distinction. Both pass WCAG AA on the card surface. Color-only discrimination is explicitly **not** an accessibility requirement — icon + label discrimination is.

**Forward-compatibility (Phase 17):** The mapping table above lives inside `RoleBadge.vue` as a stateless lookup. Phase 17 may add new enum cases by appending entries — call sites pass only `:role` and `:label` props.

## Chooser Page Layout (`/login/choose`)

**Route:** `GET /login/choose` (rendered via `ChooseRoleController::create`).
**Behavior:** Only reachable post-authentication for users holding `super_admin` + at least one tenant role. Direct navigation by single-role users redirects to their canonical home (`/admin/dashboard` or `/dashboard`).

**Visual structure (mobile-first, centered):**

```

│                                                              │
│                                                              │
│              Choose how to continue                  ← H1     │
│  You hold multiple roles on this account. Pick a    ← sub    │
│  context for this session — you can switch any time.         │
│                                                              │
│  ┌─────────────────────┐    ┌─────────────────────┐         │
│  │ [Shield icon]       │    │ [Crown icon]        │          │
│  │ Platform Admin      │    │ Owner of Test Co.   │          │
│  │ Manage tenants, …   │    │ Open your workspace…│          │
│  │                     │    │                     │          │
│  │ [Continue as PA  →] │    │ [Continue as Owner→]│          │
│  └─────────────────────┘    └─────────────────────┘          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
   min-h-screen, bg-background, flex items-center justify-center
   inner: max-w-3xl mx-auto px-4
   grid: grid-cols-1 md:grid-cols-2 gap-6 mt-8
   card outer padding: p-6
   icon size: h-8 w-8 text-primary (Owner card) / text-foreground (Admin card)
   button: w-full, size="default", variant="default", trailing ArrowRight (h-4 w-4 ml-2)
```

**Submission:**
- Each button is a `<Link>` styled via `Button asChild` — `POST /login/choose` with `{ context: 'admin' | 'tenant' }` per RESEARCH Pattern 9.
- Backend redirects to `/admin/dashboard` or `/dashboard`; from that point the URL prefix IS the active context.
- No selection persistence beyond the next request — refreshing `/admin/dashboard` does NOT re-show the chooser (the user's URL is now `/admin/*`; the layout reads the badge accordingly).

**Edge cases:**
- User has only `super_admin` AND clicks "Continue as Owner": backend rejects with 403 and flashes `You do not hold a tenant role. Contact support.` redirecting to `/admin/dashboard`. This string is added to the copywriting contract above only if implemented; planner's call on whether to render server-side (preferred) or block client-side.
- Network error during submit: existing Inertia error handling renders the error in the page-level `<Alert>` slot above the cards.

## `v-if` Sweep Visual Rules (D-14)

Every `Pages/Client/*.vue` and `Pages/Admin/*.vue` surface gains `v-if="$page.props.auth.user.can.X"` on role-gated controls. The contract for **how** the hidden controls behave visually:

| Rule | Statement |
|------|-----------|
| **R1 — Hide entirely, do not disable** | Hidden controls MUST be removed from the DOM via `v-if` (NOT `v-show`, NOT `:disabled="!can.X"`, NOT `opacity-50 pointer-events-none`). Disabled-with-tooltip variants are NOT used for ambiguous cases — D-14 is unambiguous: hide. |
| **R2 — Layout collapses** | When a control is hidden, the parent flex/grid container reflows naturally — no placeholder div, no `invisible` class, no preserved width. If a button row becomes empty, the row itself disappears (parent `v-if` on the row container if appropriate, or `:class="{ hidden: !anyVisible }"` only if reflow is undesirable). |
| **R3 — No skeleton for hidden controls** | A Manager viewing the Owner-only "Upgrade plan" button sees nothing where it would have been — no loading skeleton, no ghosted placeholder. |
| **R4 — Empty-state copy when all CRUD hidden** | If a page's ENTIRE primary action region collapses to zero visible controls for a role (e.g., a hypothetical Agent on a settings-only page), the page renders the "No actions available" empty state (copy locked above). This is **forbidden as a per-page custom string** — use the locked copy verbatim. |
| **R5 — Empty-state component** | The empty state is rendered as a centered block: `<div class="rounded-md border border-dashed bg-muted/30 p-6 text-center text-sm text-muted-foreground">` with the heading in `text-sm font-semibold text-foreground` and the body in `text-sm text-muted-foreground mt-1`. No icon. No CTA (deliberately — there is no action available). |
| **R6 — Pages with mixed-tier controls render normally** | Most pages have at least some Agent-visible controls (view, navigate). R4 only triggers when ALL action controls collapse. Lead detail page for Agent: edit + delete remain visible (Agent has `manage-leads`); no empty state. |
| **R7 — Read-only data displays are NEVER role-gated** | Tables, lists, and read panels render for any user who reached the route. Only mutation controls (buttons, form submits, "Edit"/"Delete"/"Add"/"Upgrade") gate. |
| **R8 — Forbid disabled-with-tooltip** | The pattern `<Button :disabled="!can.X" :title="'Requires Owner role'">` is explicitly forbidden by this contract. If a role distinction is hard to communicate, the answer is "hide it"; documentation lives in Phase 17's team management page, not in inline tooltips. |

## Inertia Share Contract (relevant to UI only)

The planner's `HandleInertiaRequests::share()` work must expose these flat props for the UI components above:

```typescript
auth.user: {
  id: number
  name: string
  email: string
  tenant?: { id: number; name: string } | null
  can: { [abilityName: string]: boolean }      // D-13 flat map
  has_super_admin_role: boolean                // computed in share() for ContextSwitchChip visibility
  has_tenant_role: boolean                     // computed in share() for ContextSwitchChip visibility
  primary_role: { value: string; label: string } | null
  // value ∈ { 'super_admin', 'owner', 'manager', 'agent' } | label is the locked UI string
}
```

The UI does NOT compute role priority client-side. The backend (per `HandleInertiaRequests`) resolves the active primary role using URL context (`/admin/*` → super_admin label wins; `/dashboard/*` → highest tenant role wins) and ships it as `primary_role`. This keeps Vue templates free of role logic per D-13.
