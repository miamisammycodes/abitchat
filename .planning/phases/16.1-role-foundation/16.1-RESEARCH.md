## Summary

This phase collapses the existing two-table / two-guard split (`users` + `admin_users`, `web` + `admin` guards) into a unified single-guard four-role RBAC system. All scaffolding (enum, ability enum, pivot table, RolePermissions class, Gates, middleware, policies, Inertia share, full UI sweep, role badge, dual-role chooser, seeder rebuild) ships in one phase per D-11.

Three load-bearing technical facts surfaced during research:

1. **Gate::define accepts a string only.** Per the Laravel 13 docs `[CITED: laravel.com/docs/13.x/authorization]`, the first arg is `string $ability`. Use `Ability::ManageBilling->value` (the backed enum's string value) at call sites — the Ability enum still gives compile-time safety; the Gate registry just sees strings.
2. **No `AuthServiceProvider` exists in this project.** `bootstrap/providers.php` registers only `AppServiceProvider`. Laravel 11+ idiom is to register Gates in `AppServiceProvider::boot()` — do NOT introduce a new provider. CONTEXT.md mentions `AuthServiceProvider` but the canonical project location is `AppServiceProvider`.
3. **Two foreign keys currently point at `admin_users`** — `transactions.approved_by` (migration `2026_04_27_144210`) and `admin_activity_logs.admin_user_id` (migration `2025_11_28_104711`). Dropping `admin_users` requires re-pointing both back at `users(id)` in a specific sequence — this is the single largest non-obvious migration risk.

**Primary recommendation:** Treat 16.1 as a *schema rewrite + auth-surface rewrite* phase. The migration must (1) drop two FKs, (2) drop `admin_users`, (3) re-point both FKs at `users`, (4) drop the `users.role` enum column, (5) make `users.tenant_id` nullable, (6) create `user_roles`. Use `migrate:fresh --seed` for dev — no in-place data migration is needed per D-05.

### Pattern 1: Backed PHP Enum for Roles

**What:** A string-backed enum carrying the role name + an ordinal rank for hierarchical comparison.
**When to use:** Whenever a Role value flows between DB column ↔ PHP code ↔ JSON.
**Example:**

```php
// Source: [CITED: laravel.com/docs/13.x/eloquent-mutators#enum-casting]
<?php

declare(strict_types=1);

namespace App\Enums;

enum Role: string
{
    case SuperAdmin = 'super_admin';
    case Owner      = 'owner';
    case Manager    = 'manager';
    case Agent      = 'agent';

    /**
     * Tenant-hierarchy rank. SuperAdmin is orthogonal — DO NOT compare via rank.
     * Higher = more privileged within tenant.
     */
    public function rank(): int
    {
        return match ($this) {
            self::Owner   => 3,
            self::Manager => 2,
            self::Agent   => 1,
            self::SuperAdmin => 0,  // orthogonal — never compared via rank
        };
    }

    public function isPlatformLevel(): bool
    {
        return $this === self::SuperAdmin;
    }

    public function label(): string
    {
        return match ($this) {
            self::SuperAdmin => 'Platform Admin',  // per <specifics> — never "Super Admin" in UI
            self::Owner      => 'Owner',
            self::Manager    => 'Manager',
            self::Agent      => 'Agent',
        };
    }
}
```

### Pattern 3: Canonical RolePermissions Class

```php
// Source: [ASSUMED — canonical hand-rolled pattern; verified against Laravel 13 Gate semantics]
<?php

declare(strict_types=1);

namespace App\Auth;

use App\Enums\Ability;
use App\Enums\Role;
use App\Models\Tenant;
use App\Models\User;

final class RolePermissions
{
    /**
     * Single source of truth: ability → minimum role required.
     *
     * SuperAdmin abilities require the SuperAdmin role specifically (orthogonal — not a "max rank").
     * Tenant-scoped abilities require hasRoleAtLeast() within the given tenant.
     *
     * @var array<value-of<Ability>, Role>
     */
    private const MIN_ROLE = [
        // SuperAdmin-only (exact-match)
        Ability::ViewAdminDashboard->value     => Role::SuperAdmin,
        Ability::ManageTenantAsAdmin->value    => Role::SuperAdmin,
        Ability::ManagePlatformSettings->value => Role::SuperAdmin,
        // Owner-only (tenant-scoped, exact-match within hierarchy)
        Ability::ManageBilling->value          => Role::Owner,
        Ability::ManageTeam->value             => Role::Owner,
        Ability::ManageTenantSettings->value   => Role::Owner,
        Ability::DeleteTenant->value           => Role::Owner,
        // Manager+
        Ability::ManageKnowledgeBase->value    => Role::Manager,
        Ability::ManageIntegrations->value     => Role::Manager,
        Ability::ViewAnalyticsFull->value      => Role::Manager,
        // Agent+
        Ability::ManageConversations->value    => Role::Agent,
        Ability::ManageLeads->value            => Role::Agent,
        Ability::ViewDashboard->value          => Role::Agent,
    ];

    public static function can(User $user, Ability $ability, ?Tenant $tenant = null): bool
    {
        $minRole = self::MIN_ROLE[$ability->value];

        if ($minRole === Role::SuperAdmin) {
            return $user->hasRole(Role::SuperAdmin);  // platform-level; no tenant needed
        }

        return $tenant !== null
            && $user->hasRoleAtLeast($minRole, $tenant);
    }

    /** @return list<Ability> */
    public static function allAbilities(): array
    {
        return Ability::cases();
    }
}
```

### Pattern 5: User Model Multi-Role Methods

```php
// Source: [ASSUMED — synthesized from Laravel 13 Eloquent docs + Octane safety]
public function roles(): HasMany   /** @phpstan-return HasMany<UserRole, $this> */
{
    return $this->hasMany(UserRole::class);
}

/**
 * Has the user been assigned this exact role within this tenant?
 * Platform roles (SuperAdmin) are tenant-agnostic — $tenant is ignored.
 */
public function hasRole(Role $role, ?Tenant $tenant = null): bool
{
    return $this->roles
        ->contains(fn (UserRole $ur) =>
            $ur->role === $role
            && ($role->isPlatformLevel() || $ur->tenant_id === $tenant?->id)
        );
}

/**
 * Within $tenant, does the user hold a role >= $min? Owner satisfies Manager+, etc.
 * SuperAdmin is orthogonal — does NOT satisfy tenant role tier.
 */
public function hasRoleAtLeast(Role $min, Tenant $tenant): bool
{
    return $this->roles
        ->contains(fn (UserRole $ur) =>
            !$ur->role->isPlatformLevel()
            && $ur->tenant_id === $tenant->id
            && $ur->role->rank() >= $min->rank()
        );
}
```

The `$this->roles` Collection lookup hits the lazy-loaded relationship; Eloquent caches it as a model attribute (instance-scoped, dies with the request). **No `static` memoization** — that pattern bleeds across requests in Octane workers (CONS-22-d lesson from PR #30).

### Pattern 6: HandleInertiaRequests::share() Resolved Ability Map

```php
public function share(Request $request): array
{
    $user = $request->user();
    $tenant = $user?->tenant;

    return [
        ...parent::share($request),
        'auth' => [

                ])->all(),
                'can' => $this->buildAbilityMap($user),
            ] : null,
        ],
        // 'admin' block is REMOVED entirely
        'tenant' => $tenant ? [
            'id' => $tenant->id,
            'name' => $tenant->name,
            'slug' => $tenant->slug,
        ] : null,
        // ... rest of share() unchanged ...
    ];
}

/** @return array<string, bool> */
private function buildAbilityMap(User $user): array
{
    $map = [];
    foreach (Ability::cases() as $ability) {
        // Snake_case the slug for Vue ergonomics: manage-billing → manage_billing
        $key = str_replace('-', '_', $ability->value);
        $map[$key] = Gate::forUser($user)->check($ability->value);
    }
    return $map;
}
```

Per Inertia docs `[CITED: inertiajs.com/shared-data]`, lazy props (closures) defer until accessed. The ability map runs `Gate::check()` 13 times per request — cheap (in-memory closure invocations against `$user->roles` collection) but if profiling later flags it, wrap in a closure: `'can' => fn () => $this->buildAbilityMap($user)`.

### Anti-Patterns to Avoid

- **Re-introducing the old `users.role` enum column** under any new name. The pivot is the single source of truth.
- **Adding tenant-side validation for `bot_custom_instructions`** — D-09 invariant. Add a guard test that greps every `app/Http/Requests/Client/**/*.php` and asserts none reference `bot_custom_instructions`.
- **`Gate::before()` super-admin grants-all** — breaks `auth.user.can` map completeness; super_admin abilities must be explicit.
- **`static $cached ??= ...`** memoization inside model methods — **Octane-unsafe** (process-local; bleeds across requests). Use instance properties or rely on Eloquent's relation caching.
- **Raw `where('tenant_id', ...)`** on the new `UserRole` pivot model — DEC-05 + Larastan `NoRawTenantIdWhere` rule. Use `forTenant($tenant)` via `BelongsToTenant` trait.
- **Letting any policy bypass tenant-ownership in favor of role-tier alone** — every policy must AND ownership with role-tier (D-11).
- **Introducing a new `AuthServiceProvider`** — Laravel 11+ idiom registers Gates in `AppServiceProvider::boot()`. The project has only one provider; keep it that way.
- **Lazy-loading `$user->roles` inside the `share()` hot path without instance caching** — Eloquent caches the relation after the first access, but `$request->user()` must be the same instance across the request; verify by typing it `User` and not re-fetching.
- **Pivot-table inserts without `tenant_id` in seeder** — `DatabaseSeeder` uses `WithoutModelEvents`, which **suppresses** `BelongsToTenant`'s `creating` hook. Seeder must pass `tenant_id` explicitly to every `UserRole::create([...])` call.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Authorization decision engine | Custom can/cannot helpers | Laravel `Gate` facade + `Gate::define` + `$user->can()` / `Gate::check()` | Laravel Gate is the canonical surface; integrates with `$this->authorize()`, Inertia `share`, Blade `@can`, policy auto-discovery |
| Permission storage / role management UI | Spatie/laravel-permission | A 12-line `RolePermissions` const map + a 4-column pivot table | We have a fixed `O(1)` matrix; Spatie's DB-driven permissions are an over-engineering trap for our scale |
| Role hierarchy comparator | String matching with `in_array(...)` | Backed enum with `rank()` method | Type-safety + PHPStan visibility + grep-able call sites |
| Session-cookie regeneration on role switch | Custom cookie code | `$request->session()->regenerate()` (already in LoginController) | Laravel handles fixation prevention automatically |
| Middleware error response | `response()->json(...)` hand-roll | `abort(403, '...')` | Renders correctly under both Inertia + JSON endpoints; `shouldRenderJsonWhen` in `bootstrap/app.php` already handles content negotiation |

**Key insight:** Our matrix is ~13 abilities × 4 roles. The cost of adding Spatie (~4 new tables, ~18 migrations, model-level role-assignment APIs we don't need) exceeds the cost of the 60-line `RolePermissions` class by an order of magnitude. Hand-rolled is the right call here.

## Common Pitfalls

### Pitfall 1: Migration ordering when dropping `admin_users`

**What goes wrong:** Dropping `admin_users` while `transactions.approved_by` + `admin_activity_logs.admin_user_id` still reference it = FK constraint violation. The current schema's FK to `admin_users(id)` was set by migration `2026_04_27_144210_fix_transactions_approved_by_to_admin_users.php`.

**Why it happens:** `Schema::dropIfExists('admin_users')` without first dropping dependent FKs fails on MySQL with `Cannot drop table 'admin_users' referenced by a foreign key constraint`.

**How to avoid:** Hard-code this migration sequence in the plan:

1. Drop FK `transactions_approved_by_foreign` (column kept, column data becomes orphaned)
2. Drop FK `admin_activity_logs_admin_user_id_foreign` (column kept)
3. Drop table `admin_users`
4. Add FK `transactions.approved_by → users(id) nullOnDelete()` (in a fresh `Schema::table()` call — separate from drop step on MySQL)
5. Add FK `admin_activity_logs.admin_user_id → users(id) cascadeOnDelete()` (same)
6. Drop column `users.role` (separate migration)
7. Make `users.tenant_id` nullable (separate migration)
8. Create `user_roles` table (separate migration)

Each migration's `down()` must reverse — but per D-05 + the `migrate:fresh --seed` directive, downs need not be production-tested.

**Warning signs:** Tests fail with `SQLSTATE[HY000]: General error: 1 cannot start a transaction within a transaction` or `Cannot drop table 'admin_users': foreign key constraint`.

### Pitfall 2: `AdminActivityLog::log()` reads `Auth::guard('admin')`

**What goes wrong:** Line 51 of `app/Models/AdminActivityLog.php` does `Auth::guard('admin')->user()`. When the `admin` guard is removed, this static method breaks at every call site (3+ existing call sites in the Admin controllers).

**Why it happens:** Existing code assumes the admin guard. The single-guard refactor doesn't auto-update model statics.

**How to avoid:**
```php
public static function log(string $actionType, ?Model $target = null, array $details = []): self
{
    $user = Auth::user();
    if (! $user instanceof User || ! $user->hasRole(Role::SuperAdmin)) {
        throw new \LogicException('AdminActivityLog::log called outside super_admin context');
    }
    return self::create([
        'admin_user_id' => $user->id,    // column rename below
        ...
    ]);
}
```

**Warning signs:** `Call to a member function id() on null` in `AdminActivityLog::log` during admin controller tests.

### Pitfall 3: `transactions.approved_by` + `admin_activity_logs.admin_user_id` semantics

**What goes wrong:** After re-pointing both FKs at `users(id)`, the column names are misleading (`admin_user_id` no longer points at an admin-only table).

**Why it happens:** Schema rename was scoped to drop the table, not rename the columns. Renaming dependent columns is a separate migration concern.

**How to avoid:** Surface this in Open Questions — the planner / user must decide:
- **Option A:** Keep column names as-is (`admin_user_id`, `approved_by`). Add a Larastan comment explaining "the admin who acted, identified by their super_admin role at action time."
- **Option B:** Rename `admin_user_id → user_id` (only on `admin_activity_logs`; `approved_by` already describes intent).

Recommended: **Option A.** Renaming requires touching 80+ tests/models and provides no functional benefit. The PHPDoc `@return BelongsTo<User, $this>` on the relation method is enough.

**Warning signs:** PHPStan complains about `@property AdminUser $admin` annotations after model edits.

### Pitfall 4: `DatabaseSeeder::WithoutModelEvents` suppresses `BelongsToTenant` hook

**What goes wrong:** `WithoutModelEvents` (line 17 of `DatabaseSeeder.php`) disables the `creating` event boot hook in `BelongsToTenant`. When seeder calls `UserRole::create([...])` without `tenant_id`, the row is inserted with `tenant_id = NULL` silently — which is correct for super_admin rows but a bug for tenant-scoped rows.

**Why it happens:** Existing tenant API key auto-population in `Tenant::create` hits the same trap; the seeder works around it by explicit `api_key_hash`. The same explicitness must apply to all `UserRole::create` calls.

**How to avoid:** Every `UserRole::create([...])` in the seeder MUST pass `tenant_id` explicitly (NULL for super_admin, $tenant->id for tenant roles). Verification: a unit test for the seeder that asserts `user_roles.tenant_id` matches expectations for every seeded row.

### Pitfall 5: Inertia share resolves `Gate::check()` per request — make sure `$user->roles` is loaded once

**What goes wrong:** `Gate::check()` is called 13 times in `buildAbilityMap()`. Each call's Gate closure invokes `RolePermissions::can()` → `$user->hasRoleAtLeast()` → `$this->roles->contains(...)`. If `$user->roles` isn't pre-loaded, Eloquent fires one SELECT per `Gate::check()` = 13 N+1 queries per request.

**Why it happens:** Lazy relationship loading is the Eloquent default.

**How to avoid:** Eager-load `roles` on the authenticated user inside `HandleInertiaRequests::share()`:
```php
$user = $request->user();
$user?->loadMissing('roles');
```

`loadMissing` is idempotent — won't re-query if already loaded. Single query per request, regardless of ability count.

**Warning signs:** Telescope / Debugbar shows 13+ identical `SELECT * FROM user_roles WHERE user_id = ?` queries per page load.

### Pitfall 6: PHPStan baseline regression risk

**What goes wrong:** Adding the `roles()` HasMany relation + `Role` enum cast on `UserRole` without proper docblocks pushes baseline > 0.

**Why it happens:** Larastan's templated relationships (`HasMany<X, $this>`) require explicit generic params; missing them = type error. Backed-enum casts require `@property Role $role` PHPDoc for Larastan to honor the cast.

**How to avoid:**
```php
class UserRole extends Model
{
    use BelongsToTenant;

    /** @use HasFactory<UserRoleFactory> */    // if a factory exists; if not, omit
    protected $fillable = ['user_id', 'role', 'tenant_id'];

    /** @var array<string, string> */
    protected $casts = [
        'role' => Role::class,
    ];

    /** @return BelongsTo<User, $this> */
    public function user(): BelongsTo { return $this->belongsTo(User::class); }
}

// On User.php:
/** @return HasMany<UserRole, $this> */
public function roles(): HasMany { return $this->hasMany(UserRole::class); }
```

And on User.php class-level: `/** @property-read \Illuminate\Database\Eloquent\Collection<int, UserRole> $roles */`.

**Warning signs:** `vendor/bin/phpstan analyse` adds > 0 baseline entries after the phase.

### Pitfall 7: Stripe / DK Bank webhook + widget route leak

**What goes wrong:** Stripe Cashier webhook routes (auto-registered by Cashier) and DK Bank callback routes are NOT user-authenticated. If they accidentally land inside a `Route::middleware(['auth', 'require.super_admin'])` group, they 403 every callback.

**Why it happens:** Refactoring `routes/web.php` to add `require.super_admin` to the `/admin` group can drag in routes by mistake during a rebase or merge conflict.

**How to avoid:** Verify routes/web.php structure post-edit — admin route group is tightly scoped; widget API routes are in `routes/api.php` and don't share middleware with web; Cashier routes register themselves outside `web` group.

**Warning signs:** Stripe webhook events stop being processed; DK Bank callbacks 403.

### Known Threat Patterns for Laravel + Vue Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Privilege escalation via direct route access | Elevation of Privilege | `RequireSuperAdmin` middleware on `/admin/*` group; `$this->authorize()` on every Client controller action |
| Cross-tenant data leak via stale `tenant_id` | Information Disclosure | `BelongsToTenant` trait's `creating` boot hook + Larastan rule (DEC-05) — unchanged |
| Insecure direct object reference (IDOR) on `Lead::$id` / `Conversation::$id` / `KnowledgeItem::$id` / `Transaction::$id` | Information Disclosure | Existing Eloquent policies (now stacked with role-tier per D-11) |
| Session fixation post-login | Spoofing | `regenerate()` already called in `LoginController::store` |
| Mass-assignment of `tenant_id` or `role` via request body | Tampering | `$fillable` restrictions on `User` + `UserRole`; never accept `role` or `tenant_id` from user-controlled input directly |
| Inertia leak: server resolves ability map → user reads `can.delete_tenant = true` and assumes UI-only check | Tampering | UI booleans are advisory only; backend `$this->authorize()` is canonical. Pattern is correct as long as Vue doesn't ship business logic |
| `bot_custom_instructions` prompt-injection via tenant settings | Tampering | D-09 invariant + test that greps all Client FormRequests for the field |
