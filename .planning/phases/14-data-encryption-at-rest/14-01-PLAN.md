Create `database/migrations/2026_05_20_000001_encrypt_leads_pii.php`.

Migration up() steps in order:
1. Drop the compound index: `$table->dropIndex(['tenant_id', 'email'])` — wrap in a `if (Schema::hasIndex(...))` guard for idempotency.
2. Widen columns: change name, email, phone to `mediumText()` using `$table->mediumText('email')->change()` etc. MEDIUMTEXT not TEXT — Laravel's `text()` on MySQL is 65KB, MEDIUMTEXT is 16MB, ciphertext for 64-char values is ~150 chars but use MEDIUMTEXT for headroom with APP_PREVIOUS_KEYS overhead.
3. Backfill existing rows:

```
DB::table('leads')
    ->orderBy('id')
    ->chunk(100, function ($leads) {
        foreach ($leads as $lead) {
            // Idempotency: try to decrypt — if it succeeds, row already encrypted; skip
            foreach (['name', 'email', 'phone'] as $col) {
                $val = $lead->$col;
                if ($val === null) continue;
                try {
                    Crypt::decryptString($val);
                    // Already encrypted — skip this field
                    continue;
                } catch (\Illuminate\Contracts\Encryption\DecryptException $e) {
                    // Plaintext — encrypt it
                    DB::table('leads')
                        ->where('id', $lead->id)
                        ->update([$col => Crypt::encryptString($val)]);
                }
            }
        }
    });
```

Migration down() steps:
1. Decrypt all rows back to plaintext (reverse of above: decrypt then store raw).
2. Change columns back to `string(255)`.
3. Recreate the `(tenant_id, email)` index.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API → leads table | Lead PII (name/email/phone) enters via widget API or dashboard form; encrypted before any persistence |
| DB → App layer | Crypt::decryptString using APP_KEY; if APP_KEY is compromised, all PII is compromised |
| Migration script → DB | Backfill runs at deploy time with full DB access; must not expose plaintext in logs |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01-01 | Information Disclosure | leads table at-rest | mitigate | encrypted cast — ciphertext stored, plaintext never written to DB |
| T-14-01-02 | Information Disclosure | APP_KEY compromise | accept | APP_KEY is already the trust root (JWT signing, session encryption); no additional risk introduced here |
| T-14-01-03 | Tampering | Backfill migration — ciphertext corruption | mitigate | Idempotent try/catch on decryptString; failed decrypt → re-encrypt; chunk(100) limits blast radius |
