# Domain Glossary

Terms used by the codebase and by ADRs / architecture conversations. Keep this list short and load-bearing — only add a term once the codebase actually uses it.

## Tenant
A customer account. Owns Conversations, Leads, Knowledge Items, and Usage. The platform is single-database with a `tenant_id` column on every tenant-scoped table.

## Conversation
One visitor's chat session. Holds an ordered list of Messages and accrues a lead score. Belongs to a Tenant.

## Message
A single utterance in a Conversation, either from the visitor or the assistant. Records token cost when generated by the LLM.

## Knowledge Item / Knowledge Chunk
An uploaded source (document, FAQ, webpage, raw text) and its split-up retrievable pieces. Chunks carry pgvector embeddings.

## Lead
A visitor we have contact info for, captured from a Conversation. Carries a Lead Score derived by `LeadScoringService`.

## Usage Record
A meterable event for billing — currently only token consumption per LLM call. Each row carries `tenant_id`, `type`, `quantity`, `period`, and (for token rows) the originating `conversation_id`.

## Period
A YYYY-MM string identifying a calendar month for usage aggregation (e.g. `"2026-05"`). Always derived from `now()->format('Y-m')` at write time.

## UsageTracker
The single Module that owns tenant usage: writes Usage Records, answers "how much of X has tenant T used in period P", resolves Plan/Trial limits, and busts the per-tenant usage cache on write. Lives at `app/Services/Usage/UsageTracker.php`. Every read or write of usage goes through it; direct `UsageRecord::create` or per-tenant aggregation queries should not appear elsewhere.
