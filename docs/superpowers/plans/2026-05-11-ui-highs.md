# UI-Cluster High-Severity Fixes — 2026-05-11

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close four UI-cluster HIGHs from the 2026-05-09 audit. Each is small and independent.

- **H-NEW-12** — `Client/Conversations/Index.vue` rows are click-only. `<tr @click="router.visit(...)">` with no `tabindex`, no `role`, no keydown handler. Keyboard users can't open conversations, power users can't Cmd-click or middle-click to open in a new tab. The list is the primary nav surface; this blocks accessibility.
- **H-NEW-13** — `Admin/Plans/Create.vue` slug watcher overwrites the field on every keystroke in `name`. An admin who customizes the slug after typing a name has their custom slug silently destroyed the next time they type one more character in `name`. The bug is unpredictable: the slug looks correct until the name is touched again.
- **H-NEW-14** — `Client/Billing/Plans.vue` `activateFreeTrial()` calls `trialForm.post(...)` with no `onError` handler. The backend (`Client\BillingController::activateTrial`) returns a redirect `with('error', ...)` for the "trial already used" / "already on a plan" / "plan requires payment" cases. With no handler, the form just re-enables and the user sees nothing.
- **H-NEW-15** — `Client/Billing/Plans.vue` `closeEnterpriseModal()` calls `inquiryForm.reset()` but not `inquiryForm.clearErrors()`. Reopening the modal after a validation failure shows red error messages under blank fields. Looks broken; confuses users.

**Architecture:**
- For H-NEW-12: keep the `@click` handler on the row for the bulk click target (matches existing UX) BUT add `tabindex="0"`, `role="link"`, and `@keydown.enter` / `@keydown.space.prevent` so keyboard navigation works. Also turn the "When" cell into an Inertia `<Link>` so Cmd-click / middle-click work on at least one focused element per row. This is the standard accessible-row pattern.
- For H-NEW-13: add a `slugManuallyEdited` ref. The watcher only auto-derives the slug when this is false. Set it to true on any `@input` on the slug field.
- For H-NEW-14: `HandleInertiaRequests` already shares `flash.error` and `flash.success` to Inertia's `page.props`, but `ClientLayout.vue` does **not** render a flash banner — confirmed by reading the layout. So the error is currently invisible to the user. The fix has two parts: (a) add a flash banner to `ClientLayout.vue` so any `with('error', ...)` from any client controller becomes visible (this is a per-PR layout change with site-wide impact), and (b) add `preserveScroll: true` to `trialForm.post(...)` so the banner is in view when the redirect lands. No `onError` callback is needed — Inertia auto-resets `processing` on redirect, and the layout's banner handles display.
- For H-NEW-15: add `inquiryForm.clearErrors()` to `closeEnterpriseModal()`.

**Tech Stack:** Vue 3 Composition API + Inertia 2 + Tailwind v4 + shadcn-vue. Inertia's `useForm` exposes `clearErrors()`, `reset()`, and `processing`. Test surface is mostly browser smoke (Playwright); minimal Pest coverage for the controller side of H-NEW-14.

**Branch base:** `main`. None of these files have pending changes in open PRs at the time of writing, so no rebase risk.

## Task 1: H-NEW-12 — Conversations rows are keyboard- and Cmd-click-accessible

**Files:**
- Modify: `resources/js/Pages/Client/Conversations/Index.vue` (the row markup around lines 105-119)

This task has no automated test — it's a markup change verified via browser smoke. We commit it standalone so the smoke step has a single clear deliverable.

- [ ] **Step 1: Read the current markup**

Open `resources/js/Pages/Client/Conversations/Index.vue` and locate the `<tr v-for="conv in conversations.data">` block. Confirm the current shape matches:

```vue
<tr
  v-for="conv in conversations.data"
  :key="conv.id"
  @click="router.visit(`/conversations/${conv.id}`)"
  class="cursor-pointer hover:bg-muted/40"
>
  <td class="whitespace-nowrap px-4 py-3 text-sm" :title="conv.created_at">{{ relativeTime(conv.created_at) }}</td>
  ...
</tr>
```

- [ ] **Step 2: Patch the row + first cell**

Replace the row markup with:

```vue
<tr
  v-for="conv in conversations.data"
  :key="conv.id"
  tabindex="0"
  role="link"
  :aria-label="`Conversation from ${relativeTime(conv.created_at)}, status ${conv.status}, ${conv.lead ? 'lead captured' : 'no lead'}, ${conv.messages_count} messages`"
  @click="router.visit(`/conversations/${conv.id}`)"
  @keydown.enter="router.visit(`/conversations/${conv.id}`)"
  @keydown.space.prevent="router.visit(`/conversations/${conv.id}`)"
  class="cursor-pointer hover:bg-muted/40 focus:bg-muted/40 focus:outline-none focus-visible:ring-2 focus-visible:ring-ring"
>
  <td class="whitespace-nowrap px-4 py-3 text-sm" :title="conv.created_at">
    <Link :href="`/conversations/${conv.id}`" class="hover:underline" @click.stop>
      {{ relativeTime(conv.created_at) }}
    </Link>
  </td>
  <td class="px-4 py-3 text-sm text-muted-foreground">{{ truncate(conv.latest_message?.content) || '—' }}</td>
  <td class="px-4 py-3"><span class="rounded-full px-2 py-0.5 text-xs font-medium" :class="statusBadgeClass(conv.status)">{{ conv.status }}</span></td>
  <td class="px-4 py-3 text-sm">
    <span v-if="conv.lead" class="rounded-full bg-primary/10 px-2 py-0.5 text-xs font-medium text-primary">✓ captured</span>
    <span v-else class="text-muted-foreground">—</span>
  </td>
  <td class="px-4 py-3 text-right text-sm">{{ conv.messages_count }}</td>
</tr>
```

Key changes:
1. `tabindex="0"` makes the row focusable via Tab.
2. `role="link"` tells assistive tech this acts as a link. **Note:** this also overrides the implicit `role="row"` so screen readers won't iterate cell-by-cell — that's why the `aria-label` packs in all the per-row payload (status, lead, message count) that the user would otherwise miss.
3. `aria-label` describes the row to screen readers including every status field.
4. `@keydown.enter` and `@keydown.space.prevent` activate the row from the keyboard. Space gets `.prevent` to stop the default page scroll.
5. `focus:bg-muted/40` + `focus-visible:ring-2 focus-visible:ring-ring` give a visible focus indicator.
6. The "When" cell wraps the timestamp in an Inertia `<Link>` with `@click.stop` so Cmd-click / middle-click open the conversation in a new tab without ALSO triggering the row's `@click` (which would navigate the current tab).

The `<Link>` import needs to exist. Check the script section — if it doesn't import `Link from '@inertiajs/vue3'`, add it:

```js
import { Link, router } from '@inertiajs/vue3'
```

## Task 3: H-NEW-14 — Trial activation surfaces backend errors

**Background:** `Client\BillingController::activateTrial` returns redirects with `with('error', ...)` for three failure paths (plan requires payment, trial already used, plan already active). The client-side `trialForm.post()` has no `onError` callback, so Inertia's default behavior is: redirect happens, flash arrives, but no client-side error state is set on the form. The flash IS rendered by `ClientLayout`'s shared flash component — so the message IS visible — BUT the form's `processing` state may not reset cleanly, and users may not notice the flash if it scrolls off-screen.

**Audit:** is the flash actually rendered? Check `resources/js/Layouts/ClientLayout.vue` for a flash banner. If yes, the existing UX is "user sees the flash". If no, the silent-on-error claim is fully accurate.

**Files:**
- Modify: `resources/js/Pages/Client/Billing/Plans.vue`
- Modify (potential): `resources/js/Layouts/ClientLayout.vue` (if no flash banner exists)
- Test: extend existing `tests/Feature/Client/BillingTest.php` or create one for trial activation errors

- [ ] **Step 1: Confirm the flash banner is missing**

```bash
grep -n "flash" resources/js/Layouts/ClientLayout.vue | head -10
```

Expected: no matches. `HandleInertiaRequests` shares `flash.error` / `flash.success` to `page.props.flash`, but `ClientLayout.vue` doesn't render it (verified during plan review). The fix MUST include adding a banner.

- [ ] **Step 2: Write a controller-side feature test**

Create `tests/Feature/Client/ActivateTrialErrorsTest.php`:

```php
<?php

declare(strict_types=1);

namespace Tests\Feature\Client;

use App\Models\Plan;
use Tests\TestCase;

class ActivateTrialErrorsTest extends TestCase
{
    private function makeFreePlan(): Plan
    {
        return Plan::create([
            'name' => 'Free Trial',
            'slug' => 'free-trial',
            'price' => 0,
            'billing_period' => 'monthly',
            'is_active' => true,
            'conversations_limit' => 10,
            'leads_limit' => 5,
            'tokens_limit' => 1000,
            'knowledge_items_limit' => 1,
        ]);
    }

    public function test_paid_plan_returns_flash_error(): void
    {
        $this->actingAsTenantUser();
        $plan = Plan::create([
            'name' => 'Pro',
            'slug' => 'pro',
            'price' => 500,
            'billing_period' => 'monthly',
            'is_active' => true,
            'conversations_limit' => 100,
            'leads_limit' => 50,
            'tokens_limit' => 10000,
            'knowledge_items_limit' => 5,
        ]);

        $response = $this->post(route('client.billing.activate-trial', $plan));

        $response->assertRedirect();
        $response->assertSessionHas('error', 'This plan requires payment.');
    }

    public function test_already_used_trial_returns_flash_error(): void
    {
        $this->actingAsTenantUser();
        $plan = $this->makeFreePlan();
        $this->tenant->update(['trial_activated_at' => now()]);

        $response = $this->post(route('client.billing.activate-trial', $plan));

        $response->assertRedirect();
        $response->assertSessionHas('error', 'Your free trial has already been used.');
    }

    public function test_already_on_plan_returns_flash_error(): void
    {
        $this->actingAsTenantUser();
        $plan = $this->makeFreePlan();
        $this->tenant->update([
            'plan_id' => $plan->id,
            'plan_expires_at' => now()->addMonths(1),
        ]);

        $response = $this->post(route('client.billing.activate-trial', $plan));

        $response->assertRedirect();
        $response->assertSessionHas('error', 'You already have an active plan.');
    }
}
```

These tests pin the contract: the backend produces a flash error. The client-side handler then just needs to not swallow it.
