# Bug Audit — 2026-05-09

Thorough sweep across auth/middleware, billing, LLM/knowledge, and frontend by four parallel reviewers, then top critical claims verified against source.

Excluded findings already known from the May 2026 audit (PRs #3, #4, plus the 11 documented mediums M1–M11 and the 4 lows). Anything below is new — either missed previously, or introduced by the recent fixes.

Severity = potential blast radius if exercised.

---

## CRITICAL — money / security / data integrity

### C-NEW-1 — `extendPlan` ignores `billing_period`; yearly plans get 1 month
**File:** `app/Models/Tenant.php:127-137` + `app/Http/Controllers/Admin/TransactionController.php:110`
**Verified.** `TransactionController::approve` calls `$tenant->extendPlan($plan)` with no `$months` arg. `extendPlan` defaults to `addMonths(1)` — `$plan->billing_period` is never consulted.
**Bites:** Tenant pays a full year; receives 30 days. Repeat-offender on every yearly approval.
**Fix:** Map `billing_period` inside `extendPlan` (`yearly => 12`, `monthly => 1`) and remove the parameter, or always pass it from the caller.

### C-NEW-2 — `WidgetController::regenerateApiKey` re-opens C2 (old key valid for 5 min)
**File:** `app/Http/Controllers/Client/WidgetController.php:64-70`
**Verified.** Order is `Cache::forget(old)` → `$tenant->update(new)`. Between the two calls a concurrent widget request with the old key misses the cache, hits the DB (still old key), and re-caches the tenant under the old key for 300s. The update then writes the new key but the old-key cache entry survives.
**Bites:** A leaked/rotated API key remains usable for up to 5 minutes after the user thought they revoked it. Direct regression of C2.
**Fix:** Reverse the order — `$tenant->update(...)` first (so DB authoritatively no longer maps the old key), then `Cache::forget("tenant:api_key:{$oldKey}")`.

### C-NEW-3 — `BillingController::submitPayment` accepts any `amount` ≥ 0
**File:** `app/Http/Controllers/Client/BillingController.php:88-91`
**Verified.** Validation is `'amount' => 'required|numeric|min:0'`. No comparison against `$plan->price`. A tenant can submit `amount=0`. The only safeguard is admin attentiveness during approval.
**Bites:** Free paid-plan activations whenever an admin rubber-stamps a queue.
**Fix:** `'amount' => "required|numeric|min:{$plan->price}"` (or post-validation check that `$validated['amount'] >= $plan->price`).

### C-NEW-4 — Streaming chat orphans the user message on **pre-stream** failure
**File:** `app/Http/Controllers/Api/V1/Widget/ChatController.php:218-228`
**Different from M3.** M3 covers LLM-during-stream failure; this is the same shape of bug but on the retrieval path (DB error / Ollama down before the stream opens). User message is persisted at line 218 before `retrievalService->retrieve()` at line 228 can throw.
**Bites:** Each retry inserts another orphan; `buildMessageHistory` then feeds them back to the LLM, corrupting context and inflating cost forever.
**Fix:** Either persist the user message inside the streaming closure (matches `sendMessage` at 139–143) or wrap 218-228 in a try/catch that deletes on failure.

### C-NEW-5 — `ProcessKnowledgeItem` retries duplicate every chunk
**File:** `app/Jobs/ProcessKnowledgeItem.php:62-74`
**Trigger:** Job has `tries=3`. If it crashes after `extractContent` writes chunks but before the job marks complete, retries re-extract and **append** another full set. `GenerateEmbeddings::handle` uses `whereNull('embedding')`, so all duplicates get embedded too.
**Bites:** RAG retrieval returns N copies of every chunk, skewing rank and inflating vector storage permanently.
**Fix:** Delete existing chunks before write — `$this->item->chunks()->delete();` at the top of the chunk-creation block.

### C-NEW-6 — `Analytics/Index.vue` whites out on first-time accounts
**File:** `resources/js/Pages/Client/Analytics/Index.vue:48-55`
**Trigger:** `Math.max(...props.conversationsOverTime.map(...))` and `props.leadScoreDistribution.hot` access without optional-chaining or null-guard. A tenant with no data (or a partial Inertia reload returning null) crashes the page.
**Bites:** New tenant lands on Analytics → blank page. Bad onboarding signal.
**Fix:** `(props.X ?? []).map(...)` and `props.leadScoreDistribution?.hot ?? 0` across the four time-series + the score-distribution access.

## HIGH

### H-NEW-1 — `SafeExternalUrl` lets through IPv4-mapped IPv6 addresses
**File:** `app/Rules/SafeExternalUrl.php:58-62`
`http://[::ffff:127.0.0.1]/` and `[::ffff:169.254.169.254]` pass `FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE` — those flags don't catch the v4-in-v6 form. Full SSRF, including AWS instance metadata in cloud deploys.
**Fix:** Detect the `::ffff:` prefix, extract the embedded IPv4, recheck.

### H-NEW-2 — DNS rebinding TOCTOU after `SafeExternalUrl`
**File:** `app/Rules/SafeExternalUrl.php` validation vs. `app/Services/Knowledge/DocumentProcessor.php:44` fetch
Validation does its own DNS lookup; the queued `ProcessKnowledgeItem` job does another lookup minutes later. Attacker flips DNS in between. Both individual lookups are safe, but they don't agree.
**Fix:** Resolve the host once, store the resolved IP, pin the fetch to that IP (e.g., Guzzle `CURLOPT_RESOLVE`).

### H-NEW-3 — `DocumentProcessor::extractFromUrl` follows redirects after validation
**File:** `app/Services/Knowledge/DocumentProcessor.php:44`
`Http::timeout(30)->get($url)` follows redirects by default. A validated public URL can 30x to `http://169.254.169.254/...`. Easier than DNS rebinding to exploit.
**Fix:** `Http::withOptions(['allow_redirects' => false])` — or follow but re-validate every hop.

### H-NEW-4 — `EmbeddingService` is hardcoded to Ollama
**File:** `app/Services/Knowledge/EmbeddingService.php:33`
`Provider::Ollama` is wired in regardless of env. In production (where Ollama isn't running), `generate()` returns `null` silently and retrieval falls back to keyword search. **No alarm fires.**
**Bites:** Production RAG is silently disabled; chatbot quality degrades, no signal that anything is wrong.
**Fix:** Config-driven provider/model and a hard `Log::error` (or throw) when generation fails.

### H-NEW-5 — `markAsReady` fires before embeddings exist
**File:** `app/Jobs/ProcessKnowledgeItem.php:74`
Status flips to `ready` immediately after dispatching the embedding job. `RetrievalService` uses `status='ready'` as the gate. Window between dispatch and worker pickup → widget queries hit chunks with NULL embeddings, `<=>` against null is undefined behavior in pgvector.
**Fix:** Have `GenerateEmbeddings::handle` flip the status on completion; keep `processing` until then.

### H-NEW-6 — System-prompt injection via `bot_custom_instructions` and knowledge content
**File:** `app/Services/LLM/ChatService.php:185-269`
Tenant-controlled string is concatenated into the system prompt with no delimiter/escape. Knowledge chunks (which can include scraped HTML) are injected verbatim under `## Relevant Information:`.
**Bites:** Tenant (or any compromised knowledge source) can override the operator rules — bot becomes a general-purpose assistant for that tenant's site visitors, leaks system prompt, or skips lead-capture rules.
**Fix:** XML-style delimiter wrapping operator rules; chunk content prefixed with `--- BEGIN UNTRUSTED CONTENT ---` and capped at a fixed token budget.

### H-NEW-7 — No prompt-budget guard before LLM call
**File:** `app/Http/Controllers/Api/V1/Widget/ChatController.php:139-163`
History (last 20 × ≤2000 chars) + 5 knowledge chunks + system prompt can blow Groq's context window. Result: a 422 from Groq, wrapped as `MESSAGE_ERROR` 500. User message is already persisted, retries pile up.
**Fix:** Trim history from the oldest end when estimated tokens exceed a config budget; return `CONTEXT_TOO_LONG` cleanly before billing.

### H-NEW-8 — Failed billed LLM attempts are dropped from `UsageRecord`
**File:** `app/Services/LLM/ChatService.php:65-96`
On Groq 503/429, `retry()` makes N attempts. Failed attempts can be billed by Groq but only the final successful attempt gets `trackUsage`'d. Tenants can exceed your Groq quota silently.
**Fix:** Accumulate tokens across attempts inside the retry callback.

### H-NEW-9 — `UsageTracker::remaining` treats `0` as unlimited
**File:** `app/Services/Usage/UsageTracker.php:115-123`
`if ($limit === null || $limit <= 0) return null;` — but `Plan` validation (per agent claim) allows `0` as a "block all" value, while `-1` is the explicit unlimited sentinel. So a plan with `conversations_limit=0` becomes unbounded.
**Bites:** Any plan misconfigured with a zero limit silently grants infinite usage.
**Fix:** `if ($limit === null || $limit === -1) return null;` and rely on validation to reject negatives other than `-1`.

### H-NEW-10 — `ClientController::updatePlan` collapses remaining time
**File:** `app/Http/Controllers/Admin/ClientController.php:155-171`
Saving a tenant's profile in admin with a blank `expires_at` overwrites `plan_expires_at = now()->addMonth()` — a yearly plan with 9 months left becomes a 30-day window. Silent.
**Fix:** Route through `extendPlan` (which preserves remaining time) instead of a raw `update`.

### H-NEW-11 — `TransactionController::approve` doesn't check `plan.is_active`
**File:** `app/Http/Controllers/Admin/TransactionController.php:109-111`
Tenant submits payment for plan A → admin deactivates plan A → admin approves the pending transaction. Tenant ends up subscribed to a plan that's been removed.
**Fix:** Inside the txn, after locking, assert `$locked->plan->is_active` and reject otherwise.

### H-NEW-12 — `Conversations/Index.vue` rows are click-only
**File:** `resources/js/Pages/Client/Conversations/Index.vue:106-109`
`<tr @click="router.visit(...)">` — no Enter, no Cmd-click-new-tab, no middle-click. The page is an unusable list for keyboard users.
**Fix:** Use Inertia `<Link>` (e.g., `<td><Link :href="..." class="block">...</Link></td>` or wrap-row pattern).

### H-NEW-13 — `Plans/Create.vue` slug watcher overwrites manual edits
**File:** `resources/js/Pages/Admin/Plans/Create.vue:37-39`
Watcher unconditionally derives slug from name on every keystroke — admin can't customize the slug.
**Fix:** `slugTouched` ref, only auto-derive when `!slugTouched`.

### H-NEW-14 — `Billing/Plans.vue` activateFreeTrial silent on error
**File:** `resources/js/Pages/Client/Billing/Plans.vue:30-34`
`trialForm.post()` with no `onError` and no on-page flash for failure. User clicks "Start Free Trial", server rejects, button re-enables, no visible feedback.
**Fix:** `onError` callback that surfaces flash or `form.errors`.

### H-NEW-15 — Enterprise inquiry modal keeps stale errors across opens
**File:** `resources/js/Pages/Client/Billing/Plans.vue:44-48`
`closeEnterpriseModal` calls `reset()` but not `clearErrors()`. Reopen → red text under blank fields.
**Fix:** Add `inquiryForm.clearErrors()` to `closeEnterpriseModal`.

## MEDIUM

### M-NEW-1 — Cross-transaction race on plan extension for same tenant
**File:** `app/Http/Controllers/Admin/TransactionController.php:109` + `app/Models/Tenant.php:127`
Two admins approve two pending transactions for one tenant simultaneously. Each row-locks its own Transaction, neither blocks the other; both compute `extendPlan` from the same base; second write overwrites first. Tenant pays for two months but receives one. Distinct from M2 (TOCTOU on transaction-number duplicates).
**Fix:** Take a `lockForUpdate` on the Tenant row inside the approve transaction.

### M-NEW-2 — `POST /register` has no IP rate limit
**File:** `routes/web.php:38-39`
Trial-farming and tenant-row spam from one IP. `forgot-password` POST is similarly unthrottled at route level (only a per-email broker throttle exists).
**Fix:** `->middleware('throttle:5,1')` on register; `throttle:6,1` on forgot-password POST.

### M-NEW-3 — `activateTrial` doesn't check `plan.is_active`
**File:** `app/Http/Controllers/Client/BillingController.php:157-161`
`subscribe()` aborts on inactive plans; `activateTrial` doesn't. Inconsistent and exploitable post-deactivation.
**Fix:** `abort_if(!$plan->is_active, 404);` as first line.

### M-NEW-4 — Usage cache is missed-invalidated on conversation/lead/knowledge create
**File:** `app/Services/Usage/UsageTracker.php:78-93, 140-143`
`forgetCache` is only called from `recordTokens`. Creating a conversation doesn't bust `tenant:{id}:usage`. For up to 60s after a tenant hits their conversation cap, more conversations slip through.
**Fix:** Public `invalidate(Tenant)` method, called after each create that contributes to monthly counts.

### M-NEW-5 — Usage cache key omits the period — month boundary bleed
**File:** `app/Services/Usage/UsageTracker.php:125-128, 145-148`
`tenant:{id}:usage` carries last month's totals into the first 60s of the new month, blocking tenants whose previous month was near cap.
**Fix:** `cacheKey` → `"tenant:{id}:usage:" . self::currentPeriod()`.

### M-NEW-6 — `countByPeriod` uses `whereYear`/`whereMonth` (sargability)
**File:** `app/Services/Usage/UsageTracker.php:131-138`
Function-wrapped column kills index use. At a million rows per tenant this turns each middleware check into a slow scan.
**Fix:** `whereBetween('created_at', [now()->startOfMonth(), now()->endOfMonth()])`.

### M-NEW-7 — Concurrent first messages create duplicate leads
**File:** `app/Services/Leads/LeadService.php:47-68`
Two parallel widget messages on the same conversation both see `lead_id IS NULL`, both `createLead`. One ends up orphaned; tenant gets dupe notifications.
**Fix:** Wrap in `DB::transaction` with `lockForUpdate` on the conversation row, or `updateOrCreate` keyed on `(tenant_id, conversation_id)`.

### M-NEW-8 — Vector dimension mismatch produces hard pgvector error
**File:** `app/Services/Knowledge/RetrievalService.php:49`
If embedding length ≠ column dimension, pgvector's `<=>` throws. Some Ollama builds return 384-dim from `nomic-embed-text` while column is `vector(768)`.
**Fix:** Assert `count($vector) === DIMENSIONS` in `EmbeddingService::toPgVector`; reject mismatch early.

### M-NEW-9 — `GenerateEmbeddings` loads all chunks at once
**File:** `app/Jobs/GenerateEmbeddings.php:37-45`
`->get()` materializes every NULL-embedding chunk in one go. Large PDF imports OOM the worker; tries=3 means it OOMs three times.
**Fix:** `->cursor()` or `->chunk(100)`.

### M-NEW-10 — `getTopQuestions` group-by full message body
**File:** `app/Services/Analytics/AnalyticsService.php:195-213`
`GROUP BY content` on an unindexed text column = full scan + filesort. Active tenants will gateway-time-out the analytics page.
**Fix:** Add a `content_hash` column with index, or pre-truncate before group.

### M-NEW-11 — DOCX extractor merges adjacent words
**File:** `app/Services/Knowledge/DocumentProcessor.php:79-95`
`strip_tags` on raw OOXML can produce `pricelist` from `<w:t>price</w:t><w:t>list</w:t>`. RAG silently fails on those terms.
**Fix:** Walk `<w:t>` nodes explicitly, join with a separator.

### M-NEW-12 — Plan toggle has no in-flight guard
**File:** `resources/js/Pages/Admin/Plans/Index.vue:55-59`
Double-click sends two PATCHes; final state depends on which lands last. Admin can think they deactivated when they didn't.
**Fix:** `togglingPlanId` ref; `:disabled` while in flight.

### M-NEW-13 — Admin `Dashboard.vue` and `Clients/Show.vue` unguard `stats.*`
**File:** `resources/js/Pages/Admin/Dashboard.vue:69-70`, `resources/js/Pages/Admin/Clients/Show.vue:165-166`
Optional chaining used elsewhere isn't used here. Any null in the stats payload white-screens the page.
**Fix:** `stats?.tenants?.total ?? 0` — or guarantee the shape in the controller.

### M-NEW-14 — `Conversations/Index.vue` unguarded `conversations.data`
**File:** `resources/js/Pages/Client/Conversations/Index.vue:86`
`Leads/Index.vue` uses `?.data?.length` — Conversations doesn't. Partial-reload null shapes white-screen the page.
**Fix:** `conversations?.data?.length === 0` and `conversations?.data` in the v-for.

### M-NEW-15 — `ClientLayout.vue` Settings link 404s
**File:** `resources/js/Layouts/ClientLayout.vue:186`
Goes to `/dashboard/settings`; route doesn't exist.
**Fix:** Either ship the page or remove the link.

## How to triage this list

If we ship one PR, prioritize by money-and-key impact in this order:

1. **C-NEW-1** (yearly plans short-changed) — pricing bug, real revenue.
2. **C-NEW-2** (api-key rotation cache) — security regression of a previously-fixed critical.
3. **C-NEW-3** (zero-amount payment) — revenue / approval-process risk.
4. **C-NEW-4 + C-NEW-5** (orphan + chunk dup) — data hygiene; both compound on retry.
5. **H-NEW-1, H-NEW-2, H-NEW-3** (SSRF gaps) — same family, same fix scope.

Frontend criticals (C-NEW-6) and the Conversations/Analytics null-guards (M-NEW-13/14, H-NEW-12) are cheap one-liners that should ride along.

The 11 documented Mediums from the May audit (M1–M11) are still open and not duplicated here.
